## 1. 总览概览 (Overview)

### 今日请求量

| 属性 | 说明 |
|------|------|
| **页面** | 总览 |
| **类型** | KPI |
| **计算逻辑** | `COUNT(*) FROM usage_logs ul`，筛选 `DATE(ul.created_at) = CURDATE()`，同时应用渠道/模型/用户过滤 |
| **SQL 表** | `usage_logs` |
| **业务意义** | 今日累计 API 请求总数，反映平台当前的活跃程度 |

### 日活跃用户 (DAU)

| 属性 | 说明 |
|------|------|
| **页面** | 总览 |
| **类型** | KPI |
| **计算逻辑** | `COUNT(DISTINCT ak.employee_id) FROM usage_logs ul LEFT JOIN api_keys ak ON ul.api_key_id = ak.id`，筛选今日 |
| **SQL 表** | `usage_logs` + `api_keys` |
| **业务意义** | 今日使用过平台的去重用户数，衡量平台日活规模 |

### 成功率

| 属性 | 说明 |
|------|------|
| **页面** | 总览 |
| **类型** | KPI |
| **计算逻辑** | `SUM(CASE WHEN r.status = 'completed' THEN 1 ELSE 0 END) / COUNT(*)`，统计近 30 天数据 |
| **SQL 表** | `usage_logs` + `requests` |
| **业务意义** | 近 30 天请求完成率（非 failed/canceled），反映平台整体稳定性 |

### 今日花费

| 属性 | 说明 |
|------|------|
| **页面** | 总览 |
| **类型** | KPI |
| **计算逻辑** | `COALESCE(SUM(ul.total_cost), 0) FROM usage_logs ul`，筛选今日 |
| **SQL 表** | `usage_logs` |
| **业务意义** | 今日累计 API 调用费用（美元），实时了解成本消耗 |

### 30 天总请求量

| 属性 | 说明 |
|------|------|
| **页面** | 总览 |
| **类型** | KPI |
| **计算逻辑** | `COUNT(*) FROM usage_logs ul WHERE ul.created_at >= DATE_SUB(NOW(), INTERVAL 30 DAY)` |
| **SQL 表** | `usage_logs` |
| **业务意义** | 近 30 天累计 API 请求总数，衡量平台中周期活跃程度 |

### 平均延迟

| 属性 | 说明 |
|------|------|
| **页面** | 总览 |
| **类型** | KPI |
| **计算逻辑** | `AVG(r.metrics_latency_ms)`，统计近 30 天数据 |
| **SQL 表** | `usage_logs` + `requests` |
| **业务意义** | 近 30 天请求平均延迟（毫秒），监控 AI 服务响应性能 |

### 请求量趋势

| 属性 | 说明 |
|------|------|
| **页面** | 总览 |
| **类型** | 趋势图 |
| **计算逻辑** | 按日期/周/月聚合：`DATE(ul.created_at)` 或 `DATE_FORMAT(ul.created_at, '%Y-%m-01')`，`COUNT(*)` |
| **SQL 表** | `usage_logs` |
| **业务意义** | 展示请求量的时间序列变化，发现增长或下降趋势 |

### Token 消耗趋势

| 属性 | 说明 |
|------|------|
| **页面** | 总览 |
| **类型** | 趋势图 |
| **计算逻辑** | 按时间段聚合 `SUM(prompt_tokens)` 和 `SUM(completion_tokens)`，分两线展示 |
| **SQL 表** | `usage_logs` |
| **业务意义** | 区分 Prompt 和 Completion Token 的消耗趋势，了解输入/输出占比变化 |

### 模型分布

| 属性 | 说明 |
|------|------|
| **页面** | 总览 |
| **类型** | 柱状图 |
| **计算逻辑** | 按 `model_id` 分组统计请求数，并计算每个模型的百分比占比 |
| **SQL 表** | `usage_logs` |
| **业务意义** | 展示各模型使用量排名，了解模型的受欢迎程度和分布情况 |

### 错误率趋势

| 属性 | 说明 |
|------|------|
| **页面** | 总览 |
| **类型** | 趋势图 |
| **计算逻辑** | 按时间段聚合 `SUM(CASE WHEN r.status IN ('failed', 'canceled') THEN 1 ELSE 0 END) / COUNT(*) * 100` |
| **SQL 表** | `usage_logs` + `requests` |
| **业务意义** | 错误率随时间的波动趋势，快速定位异常时间段 |

## 2. 推广覆盖度 (Adoption)

### 日活跃用户 (DAU)

| 属性 | 说明 |
|------|------|
| **页面** | 推广覆盖度 |
| **类型** | KPI（取最新时间点） |
| **计算逻辑** | DAU 趋势数据中最新一个时间点的 dau 值 |
| **SQL 表** | `usage_logs` + `api_keys` |
| **业务意义** | 直观展示最新日的活跃用户数 |

### 月活跃用户 (MAU)

| 属性 | 说明 |
|------|------|
| **页面** | 推广覆盖度 |
| **类型** | KPI（取最新时间点） |
| **计算逻辑** | DAU/MAU 趋势数据中最新一个时间点的 mau 值，粒度为月时直接按月统计，粒度为天时按 30 天滚动窗口 |
| **SQL 表** | `usage_logs` + `api_keys` |
| **业务意义** | 直观展示最新周期的月活跃用户数 |

### 活跃用户

| 属性 | 说明 |
|------|------|
| **页面** | 推广覆盖度 |
| **类型** | KPI |
| **计算逻辑** | `COUNT(DISTINCT ak.employee_id) FROM usage_logs ul LEFT JOIN api_keys ak ON ul.api_key_id = ak.id`，受时间筛选控制 |
| **SQL 表** | `usage_logs` + `api_keys` |
| **业务意义** | 筛选时间段内有过至少一次请求的去重用户数 |

### 总注册用户

| 属性 | 说明 |
|------|------|
| **页面** | 推广覆盖度 |
| **类型** | KPI |
| **计算逻辑** | `COUNT(DISTINCT employee_id) FROM api_keys WHERE employee_id IS NOT NULL` |
| **SQL 表** | `api_keys` |
| **业务意义** | 平台所有注册过的去重用户总数（基于 API Key 中的员工 ID） |

### 使用比例

| 属性 | 说明 |
|------|------|
| **页面** | 推广覆盖度 |
| **类型** | KPI |
| **计算逻辑** | 活跃用户 ÷ 总注册用户 × 100% |
| **SQL 表** | `usage_logs` + `api_keys` |
| **业务意义** | 平台渗透率的初步指标：时间段内有使用的用户占总注册用户的比例 |

### 用户渗透率

| 属性 | 说明 |
|------|------|
| **页面** | 推广覆盖度 |
| **类型** | KPI |
| **计算逻辑** | `COUNT(用户中使用天数 > 5 的用户) ÷ 总注册用户 × 100%`。分子：对 `employee_id` 分组计算 `COUNT(DISTINCT DATE(ul.created_at))`，筛选大于 5 的用户数 |
| **SQL 表** | `usage_logs` + `api_keys` |
| **业务意义** | 重度使用用户在总注册用户中的占比。使用天数 > 5 天表示用户已形成使用习惯，渗透率越高说明产品粘性越好 |

### DAU/MAU 趋势

| 属性 | 说明 |
|------|------|
| **页面** | 推广覆盖度 |
| **类型** | 折线图（双线） |
| **计算逻辑** | 天粒度：每天计算 `COUNT(DISTINCT ak.employee_id)` 作为 DAU，MAU 设为 0；月粒度：按月计算 DAU 为 0，`COUNT(DISTINCT ak.employee_id)` 作为 MAU |
| **SQL 表** | `usage_logs` + `api_keys` |
| **业务意义** | 日活和月活的时间序列对比，评估用户粘性。DAU/MAU 比值越高说明用户使用频率越高 |

### 新用户注册趋势

| 属性 | 说明 |
|------|------|
| **页面** | 推广覆盖度 |
| **类型** | 柱状图 |
| **计算逻辑** | 子查询：`SELECT ak.employee_id, MIN(ul.created_at) as first_seen FROM usage_logs ul LEFT JOIN api_keys ak... GROUP BY ak.employee_id`，按首次出现时间统计每日/月新用户数 |
| **SQL 表** | `usage_logs` + `api_keys` |
| **业务意义** | 新用户的增长趋势，反映平台的拉新速度和市场扩展情况。新用户定义为首次在 usage_logs 中出现 |

### 各渠道活跃用户

| 属性 | 说明 |
|------|------|
| **页面** | 推广覆盖度 |
| **类型** | 柱状图 |
| **计算逻辑** | 按 `ul.channel_id` 分组，`COUNT(DISTINCT ak.employee_id)` 统计每个渠道的去重用户数 |
| **SQL 表** | `usage_logs` + `api_keys` + `channels` |
| **业务意义** | 了解不同服务渠道（AI 供应商）覆盖的用户群体大小 |

### 各模型使用用户数

| 属性 | 说明 |
|------|------|
| **页面** | 推广覆盖度 |
| **类型** | 柱状图 |
| **计算逻辑** | 按 `ul.model_id` 分组，`COUNT(DISTINCT ak.employee_id)` 统计使用每个模型的去重用户数 |
| **SQL 表** | `usage_logs` + `api_keys` |
| **业务意义** | 不同模型的用户覆盖度，了解用户对模型的偏好 |

### 活跃度热力图

| 属性 | 说明 |
|------|------|
| **页面** | 推广覆盖度 |
| **类型** | 分组柱状图 |
| **计算逻辑** | `DAYOFWEEK(ul.created_at)` × `HOUR(ul.created_at)` 交叉维度，统计每个时段活跃的去重用户数 |
| **SQL 表** | `usage_logs` + `api_keys` |
| **业务意义** | 一周内每天不同时段的用户活跃分布，用于运营时段投放决策 |

### 项目活跃度排名

| 属性 | 说明 |
|------|------|
| **页面** | 推广覆盖度 |
| **类型** | 表格（Top 20） |
| **计算逻辑** | 按 `project_id` 分组，统计请求总数和去重用户数，按请求数降序排列 |
| **SQL 表** | `usage_logs` + `api_keys` + `projects` |
| **业务意义** | 各项目的活跃排名，了解哪些项目使用量最大、用户数最多 |

## 3. 深度使用 (Usage)

### 平均对话轮次

| 属性 | 说明 |
|------|------|
| **页面** | 深度使用 |
| **类型** | KPI |
| **计算逻辑** | 按 `r.trace_id` 分组统计每个 trace 中的请求数，`AVG()` 计算平均轮次，同时统计总对话数 |
| **SQL 表** | `usage_logs` + `requests` |
| **业务意义** | 用户每次对话的平均交互轮次，反映用户与 AI 的互动深度 |

### 平均 Token/请求

| 属性 | 说明 |
|------|------|
| **页面** | 深度使用 |
| **类型** | KPI（含 avg_prompt、avg_completion） |
| **计算逻辑** | `AVG(ul.prompt_tokens + ul.completion_tokens)`、`AVG(ul.prompt_tokens)`、`AVG(ul.completion_tokens)` |
| **SQL 表** | `usage_logs` |
| **业务意义** | 每次请求的平均 Token 消耗量（总量和拆分），了解使用强度的变化 |

### 流式使用比例

| 属性 | 说明 |
|------|------|
| **页面** | 深度使用 |
| **类型** | KPI |
| **计算逻辑** | `COUNT(CASE WHEN r.stream = TRUE THEN 1 END) / COUNT(*)` |
| **SQL 表** | `usage_logs` + `requests` |
| **业务意义** | 流式接口的使用占比，流式请求提供更好的用户体验（首 Token 延迟更低） |

### 会话时长分布

| 属性 | 说明 |
|------|------|
| **页面** | 深度使用 |
| **类型** | 柱状图 |
| **计算逻辑** | 按 `trace_id` 分组，`TIMESTAMPDIFF(SECOND, MIN(ul.created_at), MAX(ul.created_at))`，分段：<1m / 1-5m / 5-15m / 15-30m / 30-60m / >60m |
| **SQL 表** | `usage_logs` + `requests` |
| **业务意义** | 用户的会话时长分布，了解用户单次使用时长和使用习惯 |

### 用户请求频率分布

| 属性 | 说明 |
|------|------|
| **页面** | 深度使用 |
| **类型** | 柱状图 |
| **计算逻辑** | 按 `employee_id` 分组统计请求数，分段：1 / 2-5 / 6-20 / 21-50 / 51-100 / 100+ |
| **SQL 表** | `usage_logs` + `api_keys` |
| **业务意义** | 用户的使用频率分布，区分轻度/中度/重度用户 |

### 渠道日均请求

| 属性 | 说明 |
|------|------|
| **页面** | 深度使用 |
| **类型** | 柱状图 |
| **计算逻辑** | `COUNT(*) / COUNT(DISTINCT DATE(ul.created_at))`，按 `channel_id` 分组计算日均请求数 |
| **SQL 表** | `usage_logs` + `channels` |
| **业务意义** | 各渠道的日均请求负载，了解渠道使用压力 |

### 日人均请求数

| 属性 | 说明 |
|------|------|
| **页面** | 深度使用 |
| **类型** | 折线图 |
| **计算逻辑** | `COUNT(*) / COUNT(DISTINCT ak.employee_id)`，按时间段聚合 |
| **SQL 表** | `usage_logs` + `api_keys` |
| **业务意义** | 每个活跃用户的日均请求量，衡量用户的使用深度变化 |

### 用户留存 Cohort

| 属性 | 说明 |
|------|------|
| **页面** | 深度使用 |
| **类型** | 表格（Cohort 分析） |
| **计算逻辑** | 以用户首次出现月份为 cohort，统计后续各月的活跃用户数。通过子查询找到每个用户的 `first_seen`，再与每月活跃用户做交叉对比 |
| **SQL 表** | `usage_logs` + `api_keys` |
| **业务意义** | 用户留存率分析，了解不同时期新增用户在后续周期的留存情况 |

## 4. 产出与价值 (Value)

### 重度用户 Top 20

| 属性 | 说明 |
|------|------|
| **页面** | 产出与价值 |
| **类型** | 表格 |
| **计算逻辑** | 按 `employee_id` 分组，统计 `request_count`、`total_tokens`、`total_cost`，按请求数降序排列取前 20 |
| **SQL 表** | `usage_logs` + `api_keys` |
| **业务意义** | 使用最频繁的 Top 20 用户，用于重点客户运营和维护。同时展示其组织归属 |

### Token 排名

| 属性 | 说明 |
|------|------|
| **页面** | 产出与价值 |
| **类型** | 堆叠柱状图 |
| **计算逻辑** | 按 `model_id` 分组，分别 `SUM(prompt_tokens)` 和 `SUM(completion_tokens)` |
| **SQL 表** | `usage_logs` |
| **业务意义** | 各模型的 Token 消耗总量排名，区分 Prompt 和 Completion，了解模型资源消耗 |

### RFM 用户价值四象限

| 属性 | 说明 |
|------|------|
| **页面** | 产出与价值 |
| **类型** | 饼图 |
| **计算逻辑** | 按 `employee_id` 分组计算请求数和总费用，分四个象限：请求 ≥ 100 且费用 ≥ $10（高频高价值）、请求 ≥ 100 且费用 < $10（高频低价值）、请求 < 100 且费用 ≥ $10（低频高价值）、其余（低频低价值） |
| **SQL 表** | `usage_logs` + `api_keys` |
| **业务意义** | 基于 RFM 模型的用户分层，将用户分为四个价值象限，辅助精细化运营策略 |

### 渠道产出效率

| 属性 | 说明 |
|------|------|
| **页面** | 产出与价值 |
| **类型** | 柱状图 |
| **计算逻辑** | 按 `channel_id` 分组，`SUM(total_tokens) / SUM(total_cost)` 计算每美元获得的 Token 数（Tokens Per Dollar） |
| **SQL 表** | `usage_logs` + `channels` |
| **业务意义** | 各渠道的性价比对比，帮助成本优化决策。值越高说明单位成本的产出越大 |

### 项目贡献度趋势

| 属性 | 说明 |
|------|------|
| **页面** | 产出与价值 |
| **类型** | 折线图（多线） |
| **计算逻辑** | 按时间和 `project_id` 分组，统计每个项目在各时间段的请求数和 Token 数 |
| **SQL 表** | `usage_logs` + `projects` |
| **业务意义** | 各项目的使用量趋势对比，了解不同项目的贡献度变化 |

### 模型产出排名

| 属性 | 说明 |
|------|------|
| **页面** | 产出与价值 |
| **类型** | 柱状图 |
| **计算逻辑** | 按 `model_id` 分组，统计 `total_tokens`、`completion_tokens`、`AVG(completion_tokens)`（平均每次 Completion 产出量） |
| **SQL 表** | `usage_logs` |
| **业务意义** | 模型产出能力排名，了解哪些模型的输出量最大、每请求的平均产出最高 |

## 5. 成本分析 (Cost)

### Token + 费用趋势

| 属性 | 说明 |
|------|------|
| **页面** | 成本分析 |
| **类型** | 折线图（多线） |
| **计算逻辑** | 按时间段聚合 `SUM(prompt_tokens)`、`SUM(completion_tokens)`、`SUM(total_cost)`，该图展示前两者的趋势 |
| **SQL 表** | `usage_logs` |
| **业务意义** | Token 消耗量（分 Prompt/Completion）随时间变化，了解使用量趋势 |

### 模型成本分布

| 属性 | 说明 |
|------|------|
| **页面** | 成本分析 |
| **类型** | 柱状图 |
| **计算逻辑** | 按 `model_id` 分组，`SUM(total_cost)`、`COUNT(*)`、`SUM(total_tokens)`，按总费用降序 |
| **SQL 表** | `usage_logs` |
| **业务意义** | 各模型的成本分布排名，了解哪些模型消耗最多费用 |

### 渠道成本对比

| 属性 | 说明 |
|------|------|
| **页面** | 成本分析 |
| **类型** | 柱状图 |
| **计算逻辑** | 按 `channel_id` 分组，`SUM(total_cost)`、`COUNT(*)`、`AVG(total_cost)` |
| **SQL 表** | `usage_logs` + `channels` |
| **业务意义** | 各渠道的总体费用和单次请求平均成本，辅助渠道成本管控 |

### 成本 Top 用户

| 属性 | 说明 |
|------|------|
| **页面** | 成本分析 |
| **类型** | 表格 |
| **计算逻辑** | 按 `employee_id` 分组，`SUM(total_cost)`、`COUNT(*)`、`SUM(total_tokens)`，按总费用降序取前 20 |
| **SQL 表** | `usage_logs` + `api_keys` |
| **业务意义** | 消耗费用最高的 Top 20 用户，便于成本归因和高消耗用户管理 |

### 项目日花费

| 属性 | 说明 |
|------|------|
| **页面** | 成本分析 |
| **类型** | 折线图（多线） |
| **计算逻辑** | 按时间和 `project_id` 分组，`SUM(total_cost)` |
| **SQL 表** | `usage_logs` + `projects` |
| **业务意义** | 各项目成本的时间序列，追踪项目级别预算消耗 |

### 缓存命中率

| 属性 | 说明 |
|------|------|
| **页面** | 成本分析 |
| **类型** | 折线图 |
| **计算逻辑** | `SUM(ul.prompt_cached_tokens) / SUM(ul.prompt_tokens) * 100`，按时间段聚合 |
| **SQL 表** | `usage_logs` |
| **业务意义** | Prompt 缓存的命中率趋势。缓存命中率越高，重复 Prompt 的 Token 消耗越少，成本越低 |

### 推理 Token 占比

| 属性 | 说明 |
|------|------|
| **页面** | 成本分析 |
| **类型** | 折线图 |
| **计算逻辑** | `SUM(ul.completion_reasoning_tokens) / SUM(ul.completion_tokens) * 100`，按时间段聚合 |
| **SQL 表** | `usage_logs` |
| **业务意义** | 模型推理 Token 在 total Completion Token 中的占比趋势。推理占比高说明用户大量使用推理类模型或推理能力（如 o1、deepseek-r1），对应成本也会更高 |

### 成本预测

| 属性 | 说明 |
|------|------|
| **页面** | 成本分析 |
| **类型** | 折线图（实际+预测） |
| **计算逻辑** | 基于历史每日 `SUM(total_cost)` 数据，使用**线性回归**（最小二乘法）拟合趋势线，预测未来 7 天的成本 |
| **SQL 表** | `usage_logs` |
| **业务意义** | 成本趋势预测，辅助预算规划。斜率 = (n∑xy - ∑x∑y) / (n∑x² - (∑x)²)，截距 = (∑y - 斜率·∑x) / n |

## 通用筛选器说明

所有指标均支持以下筛选条件（影响 SQL 的 WHERE 子句）：

| 参数 | 类型 | 说明 |
|------|------|------|
| `start_date` | string | 开始日期，限制 `DATE(ul.created_at) >= :start_date` |
| `end_date` | string | 结束日期，限制 `DATE(ul.created_at) <= :end_date` |
| `granularity` | string | 聚合粒度：`day` / `week` / `month` |
| `user_ids` | string[] | 用户筛选：转换为子查询 `ul.api_key_id IN (SELECT id FROM api_keys WHERE employee_id IN (...))` |
| `channel_ids` | int[] | 渠道筛选：`ul.channel_id IN (...)` |
| `model_ids` | string[] | 模型筛选：`ul.model_id IN (...)`

## 基础数据约定

- 用户身份统一通过 `api_keys.employee_id` 识别和去重
- 总注册用户 = `COUNT(DISTINCT employee_id) FROM api_keys WHERE employee_id IS NOT NULL`
- 请求状态：`completed` = 成功，`failed` / `canceled` = 失败
- 所有时间过滤基于 `usage_logs.created_at`
- 趋势图的粒度支持 day / week / month 三档
